# 「说点AI」万人号增长策略

## 1. 目标与原则

核心里程碑是至少 10,000 名真实关注者。优先积累开发者信任、系列认知和复访习惯，不为短期数字牺牲技术准确性。

目标读者是正在实际使用 AI 编程工具、Agent 和自动化工作流的中文开发者。「说点AI」的识别点是：全栈开发者亲自测试，给出可复现结果，同时说明失败边界和真实取舍。

## 2. 内容支柱

维持下列大致比例，每月根据数据调整：

- 40%：AI / Agent 真实项目教程，负责建立信任和收藏。
- 25%：源码或工作流分析，负责建立专业识别度。
- 20%：带个人判断的工具与版本解读，负责获得新读者。
- 10%：工具对比或生态盘点，负责搜索和决策价值。
- 5%：只在有真实产品、代码或用户场景时写 AI × Web3。

不要连续发布彼此没有承接的孤立话题。优先规划 3～5 篇的小系列：入门问题→实战实现→原理深挖→失败复盘→工具选择。

## 3. 每篇文章的增长任务

每篇只设一个主任务：

| 任务 | 适合内容 | 主要观察 |
|---|---|---|
| 拉新 | 新工具、版本变化、常见痛点 | 展现、打开、新读者来源 |
| 信任 | 实战教程、源码深挖、故障复盘 | 阅读深度、收藏、有价值留言 |
| 转化 | 代表性成果、系列总结、明确方法论 | 新增关注、取关、关注理由 |
| 复访 | 系列续篇、读者问题、阶段实验 | 回访、系列文阅读、持续讨论 |

不在没有基准数据时设虚构的绝对 KPI。先连续记录至少 5 篇的基线，再对标题、开头、结构和关注引导做单变量迭代。

## 4. 长度与深度

- 快速解读：约 1800～3000 个汉字，快速回答「变了什么、影响谁、怎么做」。
- 实战教程或工作流复盘：约 3500～5500 个汉字，包含步骤、证据、原理和边界。
- 源码深挖：约 4500～7000 个汉字，沿一次真实调用或数据流展开。
- 工具盘点：按实际选择维度决定，每个工具都要有取舍，不做简介堆叠。

字数只是诊断信号。文章显得短时，依次检查：真实过程、前后对比、输出或截图、失败尝试、技术原理、选择理由、不适用场景。

## 5. 发布后复盘

只使用已提供的后台数据，不虚构缺失数字。复盘时先定位漏斗瓶颈：

1. 没有展现：检查选题时机、分发和读者匹配。
2. 有展现没点击：检查标题、封面和结果承诺。
3. 有点击没读完：检查开头、铺垫、重复和证据出现时机。
4. 读完不分享或收藏：检查是否提供可传递的观点、可复用清单或可复现方法。
5. 有阅读没关注：检查账号定位、系列承接和下一次阅读理由。

每次只调整 1～3 个可验证变量，并记录修改、预期和下次核对方式。
