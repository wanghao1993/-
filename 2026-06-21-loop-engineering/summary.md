# Loop Engineering 摘要

## 一句话总结

`Loop Engineering` 不是研究一句提示词怎么写，而是研究怎样把任务变成一个 AI 能持续推进、会验证、能回退、必要时会把决定交还给人的闭环。

## 三个关键词

### Prompt

负责沟通层。把目标、约束和输出说清楚。

### Harness

负责运行层。这里指模型外面的执行骨架，比如工具调用、权限、沙箱、日志、测试、规则文件、回滚和检查点。

### Loop

负责编排层。决定什么时候理解任务、什么时候查资料、什么时候执行、什么时候验证、什么时候停。

## 它们是不是独立的

不是互相替代，而是互相配合。

- 只有 prompt，没有 loop，更像聊天。
- 只有 harness，没有 loop，只是工具箱。
- 有 loop，但 prompt 和 harness 很弱，结果通常不稳定。

一个好用的 agent 工作流，三层都得有。

## 文章里的核心案例

文章用“给现有 SaaS 项目接入 GitHub OAuth 登录”举例，解释 loop engineering 怎么落地。

完整流程是：

1. 先把长期规则写进系统
2. 让 agent 先侦察，不急着改代码
3. 把任务拆给不同 agent 或 subagent 并行处理
4. 跑测试、看日志、做独立复查
5. 最后由人类做风险判断和签收

## 三个工具里的对应做法

### Codex

- 用 `AGENTS.md` 提供仓库级说明
- 任务在独立环境中执行
- 支持并行任务
- 返回终端日志和测试输出，方便核验

### Claude Code

- 用 `CLAUDE.md` 固定长期上下文
- 用 `hooks` 做必须执行的动作
- 用 `subagents` 做侦察和复查
- 用 `checkpoints` 回退试错

### Cursor

- 用 `.cursor/rules` 或 `AGENTS.md` 提供持久规则
- 用 `subagents` 做上下文隔离和并行分工
- 用 `checkpoints` 回退
- 用 `Bugbot` 做 PR 审查，必要时继续拉起 cloud agent 修复

## 最后想说的话

没必要因为 `Loop Engineering` 这个词焦虑。

它不是完全陌生的新东西。它更像把你本来就熟悉的工程习惯重新装回 AI 工作流：写规则、拆任务、做验证、控风险、留人工入口。

变化不是“人被替代”，而是“人从亲自敲每一步，变成设计整套流程的人”。
